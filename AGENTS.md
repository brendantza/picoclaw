# AGENTS.md - AI Coding Agent Guide for PicoClaw

This document provides essential information for AI coding agents working on the PicoClaw project.

## Project Overview

**PicoClaw** is an ultra-lightweight personal AI Assistant written in Go, inspired by [nanobot](https://github.com/HKUDS/nanobot). It is designed to run on resource-constrained devices (as low as $10 hardware with <10MB RAM) while providing a full-featured AI agent experience.

Key characteristics:
- **Ultra-lightweight**: <10MB memory footprint (99% smaller than comparable solutions)
- **Fast startup**: Boots in under 1 second even on 0.6GHz single-core processors
- **Portable**: Single self-contained binary across RISC-V, ARM, and x86 architectures
- **AI-bootstrapped**: 95% of the core codebase was generated by AI agents with human-in-the-loop refinement

Repository: https://github.com/sipeed/picoclaw
License: MIT

## Technology Stack

- **Language**: Go 1.25.7+
- **Build System**: Make + Go modules
- **CLI Framework**: Cobra (spf13/cobra)
- **Configuration**: JSON-based with environment variable support (caarlos0/env)
- **Database**: SQLite (modernc.org/sqlite) for state management
- **Logging**: Structured logging with rs/zerolog

## Build and Development Commands

### Prerequisites

- Go 1.25 or later
- Make
- golangci-lint (for linting)

### Essential Make Targets

```bash
# Build for current platform
make build

# Build for all supported platforms
make build-all

# Platform-specific builds
make build-linux-arm        # 32-bit ARM (Raspberry Pi)
make build-linux-arm64      # 64-bit ARM
make build-pi-zero          # Both 32-bit and 64-bit for Pi Zero 2 W
make build-whatsapp-native  # With native WhatsApp support

# Development workflow
make generate    # Run go generate (copies workspace)
make deps        # Download dependencies
make check       # Full check: deps + fmt + vet + test

# Code quality
make fmt         # Format code (uses golangci-lint)
make vet         # Run go vet
make lint        # Run golangci-lint
make fix         # Fix linting issues
make test        # Run tests

# Installation
make install     # Install to ~/.local/bin
make uninstall   # Remove binary
make uninstall-all  # Remove binary and all data (~/.picoclaw)

# Docker
make docker-build      # Build minimal Alpine image
make docker-build-full # Build full-featured image (Node.js 24)
make docker-run        # Run gateway
make docker-run-agent  # Run agent interactively
make docker-clean      # Clean Docker resources

# Utility
make run ARGS="--help"  # Build and run with arguments
make clean              # Remove build artifacts
make help               # Show all targets
```

### Build Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `VERSION` | git describe | Version string |
| `INSTALL_PREFIX` | ~/.local | Installation prefix |
| `PICOCLAW_HOME` | ~/.picoclaw | Workspace directory |
| `CGO_ENABLED` | 0 | Disabled for static binaries |

## Code Style Guidelines

### Linting Configuration

The project uses `golangci-lint` with a comprehensive configuration (see `.golangci.yaml`):

- **Line length**: 120 characters maximum
- **Function length**: 120 lines or 40 statements (excluding tests)
- **Cyclomatic complexity**: 20 maximum
- **Cognitive complexity**: 25 maximum
- **Import organization**: standard → default → localmodule

### Formatting Rules

```yaml
# Import order
sections:
  - standard    # stdlib
  - default     # third-party
  - localmodule # github.com/sipeed/picoclaw

# Style preferences
- Use `any` instead of `interface{}`
- Use `a[b:]` instead of `a[b:len(a)]`
- US English for spell checking
```

### Code Organization Conventions

1. **Package naming**: Short, lowercase, no underscores
2. **File naming**: snake_case for multi-word files
3. **Test files**: `*_test.go` alongside source files
4. **Build tags**: Use for platform-specific code (e.g., `//go:build linux`)

### Comment Style

- File headers should include copyright notice
- Public APIs must have documentation comments
- Use structured logging via `pkg/logger` instead of print statements

## Key Architecture Components

### 1. Agent Loop (`pkg/agent/`)

The core agent execution loop that:
- Processes incoming messages
- Manages LLM provider interactions
- Executes tools
- Handles session state and memory

Key types:
- `AgentLoop`: Main event loop
- `AgentRegistry`: Manages multiple agent configurations
- `AgentInstance`: Per-agent state and tools

### 2. Providers (`pkg/providers/`)

LLM provider abstraction supporting:
- OpenAI-compatible APIs
- Anthropic Claude
- Codex CLI
- GitHub Copilot
- Local models (Ollama, etc.)

Provider selection is configured via `config.json` with fallback chain support.

### 3. Channels (`pkg/channels/`)

Chat platform integrations implementing the `Channel` interface:
- Each channel has its own subdirectory
- Supports text, media, and interactive messages
- Auto-reconnection and error handling

### 4. Tools (`pkg/tools/`)

Extensible tool system for agent capabilities:
- `shell`: Execute shell commands
- `edit`: File editing operations
- `web`: Web search and fetch
- `spawn`: Subagent spawning
- `message`: Send messages to channels
- `filesystem`: File operations
- `mcp_tool`: MCP protocol tools
- `i2c`/`spi`: Hardware interface (Linux only)

### 5. Skills (`pkg/skills/`)

Dynamic skill system:
- Skills are installable packages from registries
- ClawHub registry support (clawhub.ai)
- Local skill loading from workspace

### 6. Configuration (`pkg/config/`)

JSON-based configuration with:
- Environment variable override support
- Migration system for config updates
- Flexible type handling (e.g., `FlexibleStringSlice`)

Configuration path: `~/.picoclaw/config.json` (override with `PICOCLAW_CONFIG`)

## Deployment

### Binary Deployment

```bash
# Build for target platform
make build-all

# Distribute appropriate binary:
# - picoclaw-linux-amd64
# - picoclaw-linux-arm64
# - picoclaw-linux-arm (32-bit)
# - picoclaw-linux-riscv64
# - picoclaw-darwin-arm64
# - picoclaw-windows-amd64.exe
```

### Docker Deployment

Two image variants:
1. **Minimal** (Alpine-based): Smaller size, basic MCP support
2. **Full** (Node.js 24): Full MCP support with Node.js tooling

```bash
# Minimal
docker compose -f docker/docker-compose.yml --profile gateway up

# Full-featured
docker compose -f docker/docker-compose.full.yml --profile gateway up
```

### Gateway Mode

The gateway runs as a long-running server:
```bash
picoclaw gateway
```

Health check endpoint: `http://localhost:18790/health`

### Agent Mode

One-shot execution:
```bash
picoclaw agent -m "Hello, world!"
```

Interactive mode:
```bash
picoclaw agent
```

## Security Considerations

- **NO CRYPTO**: PicoClaw has NO official token/coin. Any claims are scams.
- **Early Development**: Not recommended for production before v1.0
- **Network Security**: May have unresolved network security issues
- **Sandboxing**: Planned for future releases (filesystem, network)
- **Secret Management**: OAuth2 preferred over hardcoded API keys
